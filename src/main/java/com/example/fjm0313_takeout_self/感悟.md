感悟：
1. @Service和@Mapper的区别：
   （1）@Mapper注解是给接口加的，@Service注解是给实现类加的
   （2）尽管在依赖注入时都是注入的接口，但是@Service会创建实现类的实例并注入，而@Mapper在项目运行过程中自动为接口写一个实现类并注入进容器
2. 其实不用给每个entity包下的实体类设置Controller文件的，因为有些实体类是辅助类，它们或许需要设置服务接口，服务接口实现类和mapper来操作数据库数据，但是不会被单独开一个Controller;
3. 数据库中表的设计其实也有解耦的设想。以Orders，OrdersDetail，shoppingCart举例子，如果把订单中的菜品全部塞进Orders一个表中，虽然看上去容易了不少，也不用写DTO，也不用多建类，但是会有几个坏处：
   （1）单行表十分臃肿，数据库一页的数据量是固定的，单行占内存过多，就会导致分更多的页，读取的页数越多，磁盘IO也就越多
   （2）不好做查取和更改操作，更改一次的代价过于庞大，需要抓取和更新json字符串。分表设计就完全不会出这样的问题。
4. 对于高相关性的字段而言，在controller里面复用多个service是具有合理性的，因为不可能一个接口方法里写完所有对象的操作方法
5. 我的LoginInterceptor和@Transactional都体现了AOP思想，区别只是实现手段的不同。
6. 下次解决问题一定不要拿着半道上的问题去问AI，而是把发生这个异常之前自己都做了什么和相应的代码文件上传给它，有些问题其实并不是它看起来的那样，虽然错是这样报的，但是实际问题却不一定出现在这个地方
7. 在实际的业务项目中，往往不能写好一个功能再新引入一个依赖，必须得提前把依赖都规定好，不然到时候依赖很容易起兼容冲突。即使要更换依赖，也要把更改依赖前后所有的差异点全部列出来，不要改一个跑一个
8. Redis + Lua实现秒杀功能很简单，核心步骤就一个redisTemplate.execute，把Lua脚本执行一遍然后根据返回的结果确定是失败还是成功
9. RabbitMQ和RocketMQ都通常会被使用在接口里面用以快速返回结果
10. 我觉得像Rabbit MQ这种技术的出现，正是基于一种矛盾，这份矛盾源自于客户要求快速响应的体验，但是后台实际处理数据往往满足不了这份体验需求，因此就需要中间件这种东西将返回响应与处理数据这两步操作解耦，先把提示信息返回去，再在后台自己慢慢处理数据。
11. 不是所有的service的方法都要暴露给controller，有些拱后台自己实现的方法就不需要暴露给controller
