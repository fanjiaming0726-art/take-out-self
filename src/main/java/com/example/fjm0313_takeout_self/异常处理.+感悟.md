1. 测试接口时返回的是[]而不是json字符串
排查：发现是Result类未加@Data注解，导致序列化为json的时候，无法调用Result的getter方法赋值
解决：使用@Data注解
2. NoSuchBeanDefinition异常
排查：这是bean对象未注册进容器的异常，通过报错信息得知是“com.example.fjm0313_takeout_self.mapper.EmployeeMapper”没有被注册进容器，检查发现是因为没有加@Mapper注解
解决：使用@Mapper注解
3. 明明登录成功，测试current接口的时候，却返回未登录提示信息
排查：这是由于setAttribute时，使用了user.getId(),而user只含有name和password两种信息，导致id为空，通过空id查出了null
解决：更换为实际数据库的用户id


@Service和@Mapper的区别
1：@Mapper注解是给接口加的，@Service注解是给实现类加的
2：尽管在依赖注入时都是注入的接口，但是@Service会创建实现类的实例并注入，而@Mapper在项目运行过程中自动为接口写一个实现类并注入进容器



思考：
1. 为什么针对菜品的口味，份量需要单独再建一张表，一个新实体类，为什么不能直接一个dish表，一个dish实体类搞定所有问题
答案：因为口味。份量这种字段属于是一对多的关系，如果在一列当中全部体现，看似比较方便，但不方便后续的提取，修改。所以针对新字段，可以新建一张菜品偏好表，用dish_id保持与原先菜品的联系，用id来区分到底是微辣还是中辣，是大份还是小份。如果没有dish_id的话，是不知道针对哪个菜品选口味的 
2. HTTPServletRequest是根据前端传过来的字节流/json字符串映射到Tomcat服务器的吗？
答案：是，HTTPServletRequest是Tomcat服务器根据前端传过来的请求信息字节流，将字节流信息打包成HTTPServletRequest的。我们需要用就在括号里面写参数，不需要就不写，但不管需不需要Tomcat服务器都会自动生成
3. User的logout方法是不是有问题，真实情况下肯定有多个用户登录，单个用户移除了userId，对于其他用户不会产生影响吗？
答案：不会，因为每一个用户获得的session是独立的
